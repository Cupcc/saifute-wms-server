# 销售项目主题需求

## Metadata

- ID: `domain-sales-project-management`
- Status: `confirmed`
- Scope: `domain-level`
- 状态说明: 当前运行时已形成独立 `src/modules/sales-project` 与 `sales_project*` 逻辑模型；`2026-05-14` 已确认并收口销售项目物料不是目标清单 / 计划量，而是通过真实入库、销售出库和销售退货单据沉淀出来的项目归属实际库存。`F2/F3` 已按项目归属实际库存、项目价格层和项目来源层 FIFO 口径完成修正，`F5` 普通库存转项目 / 项目预留继续作为后续阶段能力保留。

## 主题定义

- `sales-project` / `salesProject` 专指对外销售的大型项目，不包含内部 `RD`（Research and Development）主题。
- 本主题解决“哪些真实库存已经归属某个销售项目、项目名下当前还有哪些价格层可出库、已经向客户真实发出了哪些原料 / 半成品 / 成品、退回了多少”的问题。
- 销售项目不是项目管理系统，不承担排期、甘特图、审批流、任务协同或工程进度管理。
- 销售项目不是新的物理仓库或 `stockScope`，但销售项目归属库存是实际库存归属，不是计划数；真实库存增加继续来自 `inbound`，真实库存减少继续来自 `sales`。
- 本主题后续可选支持“普通库存转项目 / 项目预留”语义，但入库时直接选择销售项目不是预留，而是项目归属实际入库。
- 本文档只保留长期约束、长期业务口径、能力清单和能力合同；单次交付直接从本 domain 创建 `task-*.md`。

## 承接的项目级能力版图

### 销售项目（`sales-project`）

- 本 domain 承接项目总纲中 `sales-project` 模块的长期职责，作为销售项目主档、项目归属实际库存视图、项目维度发货统计和可选库存归属调整 / 预留语义的真源。
- 项目页面需要作为独立入口存在，供仓库管理员或业务人员维护销售项目主档、查看项目归属物料、查看项目实际库存 / 项目价格层 / 已发货 / 已退货情况，并在需要时快速发起与项目相关的销售出库草稿。
- 销售项目本身不直接写库存；项目页上的“一键出库”本质上应生成 `sales` 域的标准销售出库单草稿或正式单据，而不是让项目模块旁路改库存。
- 后续若启用普通库存转项目 / 项目预留，也必须形成独立记录，不得被项目视图或项目出库草稿偷换成真实出库；但项目验收入库本身已经是实际库存归属。

### 相邻共享边界

- `sales` 继续承接销售出库 / 销售退货，并作为销售项目真实库存流出的唯一下游业务入口；项目相关统计默认复用 `sales` 单据事实。
- `inbound` 继续承接真实入库；项目验收 / 项目收货必须通过 `inbound` 单据绑定 `salesProjectId` 或等价项目快照形成项目归属库存，销售项目不拥有入库主单据。
- `inventory-core` 继续作为所有库存变动的唯一写入口；项目域只组织项目语义、项目视图、统计和后续库存归属调整 / 预留规则，不旁路改库存。
- `rd-subwarehouse` 是独立内部研发主题，不与销售项目强绑定；`RD` 物料流转与小仓协同不应再借用销售项目语义。

## 长期约束

- `C1` 不是项目管理系统：销售项目域只承接主档、项目归属库存视图、项目维度统计和后续可选库存归属调整 / 预留，不引入排期、阶段、任务协同、里程碑或审批流。状态：`生效中`
- `C2` 销售项目与 `RD` 分离：销售项目不与 `RD` 小仓、研发采购、研发领退报废建立默认强关联。状态：`生效中`
- `C3` 项目不是物理仓库：销售项目不形成新的 `stockScope`，但项目归属库存是 `inventory_log.projectTargetId` 等项目归属事实上的实际库存。状态：`生效中`
- `C4` 真实库存变化只走标准单据：项目归属库存增加只由绑定销售项目的 `inbound` 等真实入库单据产生，库存减少只由绑定销售项目且消耗该项目来源层的 `sales` 出库单产生；销售项目本身不直接过账库存。状态：`生效中`
- `C5` 项目维度真实发货口径：凡属于销售项目的真实供货，必须在销售出库单行上绑定 `salesProjectId` 或等价销售项目快照；不能只在报表层事后猜测归属。状态：`生效中`
- `C5a` 项目出库来源约束：销售项目出库不只是在出库行打项目标签，还必须在成本来源分配时限制为当前项目归属的来源层；不得用普通库存或其他项目库存补项目出库。状态：`生效中`
- `C6` 项目库存归属调整 / 预留不等于出库：若后续支持把普通库存改归属到项目或先预留给项目，必须形成独立记录，可释放、可转出库、可追溯，但不得伪造成销售出库。状态：`生效中`
- `C7` 项目可覆盖多类物料：销售项目允许统计流向客户的原料、半成品、成品，不把销售出库限制为成品。状态：`生效中`
- `C8` 当前 RD 运行时不反向定义需求：现有 `rd-project` 运行时及其 `rd_project*` 物理表不能继续作为销售项目需求口径的依据。状态：`生效中`

## 长期业务口径

### 项目主档口径

- 销售项目主档第一阶段至少需要支撑：建档、查询、修改、删除 / 作废、项目实际库存查看、项目发货统计查看。
- 项目主档字段优先围绕供货与统计需求设计，例如：`salesProjectCode`、`salesProjectName`、关联客户、项目负责人、业务日期、备注，以及可选合同 / 订单参考信息。
- 未发生任何下游引用的项目允许轻量删除；一旦已经关联验收入库、销售出库、销售退货、库存归属调整 / 预留或其他下游事实，则只能保留历史，不做静默硬删。
- 项目主档不是库存单据，也不是销售订单主单据；它只提供项目归属上下文，不维护目标物料清单或计划数量。

### 项目实际库存视图与供货口径

- 销售项目没有独立物理仓库或 `stockScope`；项目页看到的“项目库存”是按项目归属事实聚合出来的实际库存视图，核心来源是 `inbound` / `sales` / `inventory-core` 真实单据和库存流水。
- 项目物料不是项目主档上的目标清单。只有通过项目验收 / 项目收货、销售退货回补或后续明确的库存归属调整进入项目归属的物料，才构成销售项目实际库存。
- 项目库存视图至少需要回答：项目名下当前有哪些物料、各物料有哪些成本价格层、每个价格层可用多少、累计入库多少、累计出库多少、累计退货多少、当前项目归属库存多少。
- 项目名下库存仍在主仓等真实仓别中，但必须通过 `projectTargetId` 或等价项目归属字段与普通库存、其他项目库存区分。
- 项目出库草稿和正式销售出库必须只从当前项目归属来源层中选择价格层；如果一个物料存在多个项目价格层，用户必须选择具体成本价层，后台在该项目价层内 FIFO 分配来源。
- “待供货”不再以项目目标量推导；若未来需要合同数量、订单数量或客户承诺数量，必须作为销售合同 / 订单类输入单独定义，不能把项目实际库存视图伪装成计划清单。

### 项目库存归属调整 / 预留口径

- 入库时选择销售项目不是“预留”，而是项目归属实际入库：库存真实增加，来源层带项目归属，后续项目出库只能消耗这些项目来源层。
- 若后续需要把已存在的普通库存改归属到某销售项目，必须作为“库存归属调整”独立建模，保留来源、数量、价格层、原归属、目标项目、创建人和审核 / 释放证据。
- 若后续需要“预留但暂不改归属”，也必须形成独立预留记录；预留不减少 `quantityOnHand`，也不等同销售出库。
- 已归属项目的库存转成真实供货时，必须生成 `sales` 出库单；预留或归属调整本身不能伪造成客户发货。

### 项目出库与统计口径

- 销售项目的真实供货统一通过 `sales` 出库单完成；项目页上的“一键出库”应理解为“按当前项目实际库存和价格层生成销售出库单草稿并带出物料”，而不是直接扣减库存。
- 项目真实统计以销售出库 / 销售退货为准，至少应支持：累计出库数量、累计退货数量、净发货数量、出库金额、退货金额、净发货金额、出库成本、净发货成本。
- 项目统计的核心关注点是“流向客户的物料”，允许覆盖原料、半成品、成品；不强制要求先经过成品化再出库。
- 若一个项目分多批供货，项目页面必须能同时看到“项目当前库存”和“已发货 / 已退货 / 净发货”部分，而不是只有项目总量。

### 查询与角色口径

- 仓库管理员需要把销售项目列表和项目详情作为日常入口，用于查看项目主档、项目相关库存视图和项目发货统计。
- 仓库管理员在验收入库时需要能选择销售项目，使来料直接进入项目归属库存。
- 业务或销售协同人员若后续接入，可主要消费项目实际库存、项目价格层和已发货汇总，不直接改库存。
- `sales` 域用户 / 页面在录入出库时需要能选择项目，保证真实发货数据从单据入口就带上项目维度。
- `RD` 小仓管理员不应因为系统里存在销售项目而默认进入项目流程；两类主题的入口、口径和统计应保持分离。

## 能力合同（推荐）

### `F1` 销售项目主档轻量 CRUD

- In scope:
  - 维护项目编码、项目名称、关联客户、负责人、业务日期、备注等轻量主档字段。
  - 提供项目新建、查询、修改、删除 / 作废的基础能力。
  - 形成稳定的销售项目引用，供 `sales` 出库单和项目统计使用。
- Out of scope / non-goals:
  - 不引入 `立项 / 进行中 / 暂停 / 关闭` 这类项目管理状态机。
  - 不引入甘特图、任务协同、审批流和多层项目结构。
  - 不把项目主档扩展成库存单据或销售出库主单据。
  - 不在项目主档中维护目标物料清单、计划数量或待供货目标量。
- Completion criteria:
  - `[TC-1]` 文档明确销售项目主档只提供业务上下文，不直接改库存，也不承载目标物料清单。
  - `[TC-2]` 文档明确 `salesProjectCode` 在系统内唯一，且项目名称 / 客户 / 负责人等字段用于后续出库和统计引用。
  - `[TC-3]` 文档明确已被验收入库、销售出库、销售退货或库存归属调整 / 预留引用的项目不能静默硬删。
- Evidence expectation:
  - domain 文档真源 + 项目主档 API / 数据模型 / 权限收口的交付证据。
- Default derived slice acceptance mode: `full`

### `F2` 项目归属实际库存 / 价格层视图

- In scope:
  - 项目详情按物料和成本价格层查看项目归属实际库存、已发货、已退货、净发货等核心视图。
  - 支持按项目归属库存查看原料、半成品、成品三类物料的供货状态。
  - 视图优先基于带 `projectTargetId` 的真实库存流水、销售出库 / 销售退货事实派生。
  - 入库 / 验收单绑定销售项目后，其库存来源层必须进入项目实际库存视图。
- Out of scope / non-goals:
  - 不引入第二套“项目库存余额主表”与 `inventory-core` 并行记账。
  - 不把项目主档上的目标数量或计划清单当成项目库存来源。
  - 不用主仓普通库存或其他项目库存伪装为当前项目可出库库存。
- Completion criteria:
  - `[TC-1]` 文档明确项目库存视图是按项目归属事实聚合的实际库存视图，不是新的 `stockScope`。
  - `[TC-2]` 文档明确项目页至少能区分项目当前库存、项目价格层、已发货、已退货和净发货。
  - `[TC-3]` 文档明确原料、半成品、成品都可进入项目供货视图，而不只限成品。
  - `[TC-4]` 文档明确项目库存来源必须是绑定销售项目的真实入库 / 退货 / 归属调整事实，不允许由项目目标清单推导。
  - `[TC-5]` 文档明确项目出库价格层必须按当前项目归属来源层过滤。
- Evidence expectation:
  - domain 文档真源 + 项目详情查询接口 / 报表原型或交付证据。
- Default derived slice acceptance mode: `light`

### `F3` 项目关联销售出库与一键生成草稿

- In scope:
  - `sales` 出库单行支持绑定项目。
  - 项目页可按当前项目实际库存和价格层选择物料并生成销售出库草稿，减少重复录入。
  - 项目出库必须只消耗当前项目归属来源层。
- Out of scope / non-goals:
  - 不允许项目页面直接扣减库存。
  - 不绕过 `sales` 家族新增一套项目出库单。
- Completion criteria:
  - `[TC-1]` 文档明确销售项目的真实库存减少只由销售出库单产生。
  - `[TC-2]` 文档明确项目页上的“一键出库”后台必须沉淀为 `sales` 出库草稿或正式单据，而不是直接改库存。
  - `[TC-3]` 文档明确出库单行上的 `salesProjectId` 与项目快照是项目维度真实统计的基础事实。
  - `[TC-4]` 文档明确项目出库的成本来源分配必须限制在当前项目归属来源层，不能只给出库行打项目标签。
- Evidence expectation:
  - domain 文档真源 + `sales` / 项目集成 task 的接口与页面验证。
- Default derived slice acceptance mode: `full`

### `F4` 项目维度发货 / 退货 / 净发货统计

- In scope:
  - 提供按项目、按物料的发货统计视图，至少覆盖出库数量、退货数量、净发货数量、金额与成本。
  - 统计默认来自 `sales` 出库 / 退货事实，不旁路自己维护平行账。
  - 支持按原料、半成品、成品查看项目供货结果。
- Out of scope / non-goals:
  - 不把项目统计扩展为总账、月结或标准成本系统。
  - 不要求本能力单独定义全部导出模板和财务报表格式。
- Completion criteria:
  - `[TC-1]` 文档明确项目统计聚焦“流向客户的物料”，而不是 `RD` 领退报废。
  - `[TC-2]` 文档明确项目统计至少区分出库、退货和净发货三类结果。
  - `[TC-3]` 文档明确项目金额 / 成本汇总来自真实客户单据及其来源成本结果，不能靠静态估算替代。
  - `[TC-4]` 文档明确项目统计、报表和月报应继续复用 `sales` 家族真实事实，不另建平行项目发货台账。
- Evidence expectation:
  - domain 文档真源 + 项目统计页 / 月报联动验证。
- Default derived slice acceptance mode: `full`

### `F5` 项目库存归属调整 / 预留

- In scope:
  - 建立独立的项目库存归属调整或预留记录，表达普通库存改归属到项目或优先留给项目使用。
  - 支持项目页快捷发起普通库存转项目或预留。
  - 支持释放、改归属、部分转销售出库，并保留追溯关系。
- Out of scope / non-goals:
  - 不把项目库存归属调整 / 预留直接记为真实出库。
  - 不允许预留直接改写 `quantityOnHand`。
  - 不把入库时直接选择销售项目误判为预留；那是项目归属实际入库。
  - 不要求 V1 就完成复杂 ATP、锁货优先级冲突引擎或多项目智能抢占。
- Completion criteria:
  - `[TC-1]` 文档明确项目库存归属调整 / 预留是独立记录，不等同真实出库。
  - `[TC-2]` 文档明确预留创建后，库存在手数量不变；归属调整只改变项目归属事实，不产生客户发货。
  - `[TC-3]` 文档明确归属调整 / 预留可释放、可改归属、可部分转销售出库，并保留追溯证据。
  - `[TC-4]` 文档明确项目验收入库直接形成项目实际库存，不需要再生成一条预留记录。
- Evidence expectation:
  - domain 文档真源 + 后续项目库存归属调整 / 预留 task 的数据模型与交互验证。
- Default derived slice acceptance mode: `full`

| --- | --- | --- |

## 阶段交付基线

### Phase 1（`F1` + `F2`）

- Phase 1 的交付目标是先形成稳定的销售项目主档和项目详情读模型，让仓库管理员可以按项目查看项目归属实际库存、项目价格层和基础供货结果，而不把项目主档做成目标清单。
- Phase 1 至少要收敛两件事：一是稳定的项目引用轴（项目编码、项目名称、客户、负责人、业务日期、备注等），二是围绕项目归属库存的视图解释口径，明确项目库存来自真实库存流水归属，不是新的 `stockScope`，也不是计划数量。
- 项目页面中的项目库存、已发货 / 已退货结果必须消费已经存在的销售项目归属事实；如果上游真实事实尚未形成，这部分允许先表现为空结果或只读占位，但不能为了补页面展示另建一套平行项目发货账或目标清单。
- Phase 1 不包含项目页直接生成销售出库单，不包含普通库存转项目 / 预留写模型，也不把“分配到项目”提前做成锁货引擎；这些都不是本阶段成立前提。

### Phase 2（`F3` + `F4`）

- Phase 2 的交付目标是打通“项目归属实际库存 -> `sales` 出库 / 退货事实 -> 项目统计”的正式闭环，使销售项目不只停留在读模型聚合，而是具备稳定的真实供货绑定、项目来源层消耗和统计来源。
- Phase 2 至少要收敛三件事：一是 `sales` 出库单行正式绑定 `salesProjectId` 与项目快照，二是项目页按当前项目实际库存和价格层生成销售出库草稿，三是项目维度发货 / 退货 / 净发货数量、金额和成本统计全部复用 `sales` 家族真实事实。
- Phase 2 完成后，Phase 1 中“项目库存 / 已发货 / 已退货 / 净发货”的读模型应以 `inbound` / `sales` 家族真实单据事实为主源，不再依赖临时补录、目标清单或外部汇总口径。
- Phase 2 仍不允许项目模块直接写库存，不新增项目专属出库单，也不把库存归属调整 / 预留、严格锁货、ATP 或多项目抢占提前混入本阶段。

## 已确认补充口径（2026-04-10）

- 销售项目与 `RD` 分开，二者不再共享同一套业务定义。
- 销售项目统计的对象是流向客户的物料，允许覆盖原料、半成品、成品。
- 销售项目的真实库存变动统一走 `sales` 出库 / 退货，不再定义项目自有库存过账动作。
- Phase 1 的已发货 / 已退货结果默认消费既有销售项目归属事实；若 Phase 2 尚未打通正式写路径，不允许为补页面展示新增平行项目发货账。
- Phase 2 的核心是把项目上下文正式带入 `sales` 单据和项目统计，不把库存归属调整 / 预留前置为本阶段成立前提。
- 库存归属调整 / 预留若后续落地，仍必须以独立记录存在，不得回退为项目页直接改库存或把入库快捷操作伪装成真实销售出库。

## 已确认补充口径（2026-05-14）

- 销售项目没有“目标清单 / 计划物料清单”这一主线语义；项目物料必须来自真实单据沉淀出来的项目归属实际库存。
- 来了一批物料，如果业务上确认属于某销售项目，应在验收 / 入库单据中直接选择销售项目，使入库流水带项目归属；这批库存仍在主仓等真实仓别中，但业务归属为该销售项目。
- 项目详情页应围绕项目归属实际库存展示物料、价格层、项目当前库存、累计出库、累计退货和净发货；不再以目标数量推导待供货。
- 项目出库必须只消耗当前项目归属来源层。实现上不能只写出库行 `salesProjectId`，还要确保库存来源分配限制到当前项目的 `projectTargetId`。
- `F2/F3` 修正证据：项目详情读模型按 `projectTargetId` 聚合项目实际库存和价格层；项目出库草稿带出 `selectedUnitCost`；正式销售出库创建 / 修改时同时传入 `projectTargetId` 与 `sourceProjectTargetId`，使 FIFO 来源分配限制在当前项目归属来源层。
- 若未来需要合同数量、订单数量、客户承诺数量或未供数量，必须作为销售合同 / 订单能力单独建模，不能回退成项目主档上的目标物料清单。

## 待确认

- 项目验收单前端入口是从 `/sales/project` 详情页发起为主，还是从验收单页面选择销售项目为主；建议两边都支持，第一版以验收单选择项目为硬入口。
- 项目价格层可用量是否需要在普通库存价格层之外单独展示为“项目价格层”；建议单独展示，避免用户把主仓普通库存误认为项目库存。
- 普通库存转项目是否需要第一版实现；若需要，必须作为 `F5` 独立 task，不混入项目验收入库。
- 一个销售出库单行是否允许同时绑定项目和其他业务维度，例如销售订单号或合同号。
