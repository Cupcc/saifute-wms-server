## 用户需求

- [x] 系统需要在同一软件平台内同时支撑主仓与研发小仓协同作业，不要求拆成两套系统。
- [x] 研发小仓在业务上按真实独立运转的小仓处理，而不是简单的研发领用标签或项目视图。
- [x] 第一阶段研发小仓只与主仓对接；主仓发料/调拨后，系统应自动入库到小仓，不要求小仓管理员二次确认收货。
- [x] 货物进入公司、离开公司仍统一经过主仓；研发小仓第一阶段不直接承接外部采购入库、外部退回入库或对外退货。
- [x] 小仓管理员只关注自己小仓的信息与操作，需要提供面向小仓的独立工作台，并对主仓运营面做隔离。
- [x] 隔离采用折中模式：主操作台、数据、报表以小仓为主，但允许保留少量公共能力，如个人中心/登录态、物料只读选择、研发项目或归集项只读选择。
- [x] 小仓管理员第一阶段至少需要支持：查看小仓库存与流水、查看主仓调拨后自动入小仓结果、在小仓执行出库/领用、执行盘点/库存调整、执行本仓报废、查看仅属于小仓的报表。
- [x] 小仓管理员不负责退回主仓或对外退货等公司级二次流转动作，以减少额外操作负担；相关退料/退货仍由主仓统一处理。
- [x] 小仓出库原则上必须关联研发项目；若为办公用品等通用消耗，也需要落到明确的项目式归集项。
- [x] 研发小仓可以提出采购需求，由小仓管理员录入采购信息，再由采购角色执行采购。
- [x] 研发采购物料必须关联研发项目或项目式归集项，不能脱离项目单独采购。
- [x] 主仓需要可见研发采购信息，以便对照到货实物完成验收核对。
- [x] 主仓填写验收单时应能直接选择对应研发采购信息并自动带出内容，避免重复手工填写。
- [x] 研发采购到货后应先由主仓验收入主仓；考虑到主仓发料到小仓存在时间差，库存不应在主仓验收时直接记入小仓，而应在实际交接时再转入小仓。
- [x] 研发采购链路中的物料需要具备独立状态跟踪，且该状态不等同于采购单、验收单或领料单自身的审核状态。
- [x] 研发采购链路中的物料状态至少包括：`待采购`（填写采购表单保存后的默认状态）、`采购中`（采购角色确认接单后）、`取消`（被采购驳回或其他原因终止）、`到货`（实物到达主仓）、`验收`（已转入小仓）、`领取`（已被项目领用）、`报废`、`退回`。
- [x] 其中 `到货` 明确指到达主仓，`验收` 明确指完成主仓到小仓的交接；`退回` 属于公司级流转结果，仍由主仓统一执行，不要求小仓管理员直接操作退回动作。
- [x] 正式建模方向优先采用“受限子仓模型”：承认主仓与研发小仓的仓别协同，但第一阶段不扩展为完整开放式多仓体系。

## 当前进展

- 阶段进度: `Slice 1` 已实现并复审通过：研发独立工作台、固定仓别读写范围、小仓库存/流水/自动入库结果查看、项目领用、本仓报废，以及研发侧分类报表已落地，并完成浏览器冒烟验证。
- 当前状态: 本需求已从“仅确认口径”推进到“已完成首个 operating foundation 切片”；当前正式口径仍为“同平台内主仓 + 研发小仓受限协同，小仓真实独立运转，但第一阶段只与主仓对接”，其中主仓到 RD 自动交接、采购链路、物料状态链、盘点/调整仍属后续切片。
- 已确认口径 1: 小仓管理员第一阶段默认能力为库存/流水查看、自动入库结果查看、小仓出库/领用、盘点/库存调整、本仓报废，以及仅小仓报表。
- 已确认口径 2: 小仓出库必须绑定研发项目或项目式归集项，公司级入出与退回动作仍统一由主仓承担。
- 已确认口径 3: 研发采购由小仓提出需求、采购角色执行、主仓按采购信息验收，且验收单可直接选取采购信息自动带出内容。
- 已确认口径 4: 研发采购到货后先入主仓，再按真实交接时点转入小仓，不在主仓验收时直接记入小仓库存。
- 已确认口径 5: 填写采购信息保存后默认 `待采购`；采购角色点击确认接单后切换为 `采购中`；状态流还须覆盖 `取消`、`到货`、`验收`、`领取`、`报废`、`退回`，其中 `到货=到主仓`、`验收=转入小仓`。
- 口径细化: `2026-03-26` 将采购前置拆分为「表单默认待采购 → 采购确认后采购中」，与 已确认口径 5 一致。
- 阻塞项: None
- 下一步: 后续若继续推进，需另开新切片承接主仓到 RD 自动交接、研发采购需求与主仓验收联动、研发物料独立状态链，以及小仓盘点/库存调整能力。
