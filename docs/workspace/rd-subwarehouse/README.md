## 初步前提

1. 研发小仓是一个真实独立运转的小仓，不是单纯的研发领用标签。
2. 货物进入公司、离开公司仍统一经过主仓；研发小仓第一阶段只与主仓对接。
3. 主仓到小仓的流转应由主仓单据自动过账到小仓，不增加小仓二次确认负担。
4. 小仓的出库、盘点、报废都属于小仓内部动作，但出库必须绑定研发项目或项目式归集项。
5. 小仓管理员需要独立工作台，但只保留极少数不暴露主仓运营面的公共能力。
6. 研发采购需求由小仓侧提出、采购角色执行、主仓按采购信息验收入库，采购与验收之间应建立可选取、可追溯的单据关联。
7. 研发采购到货后应先入主仓，再按真实交接时点转入小仓；若存在时间差，不应在主仓验收时直接记入小仓库存。
8. 研发采购链路中的物料需要维护独立状态流，至少覆盖 `待采购 -> 采购中 -> 到货 -> 验收 -> 领取/报废/退回`，并允许在前置阶段进入 `取消`。
9. 所有库存结果仍必须经过 `inventory-core`，不能绕过库存核心直接改数。

## 研发物料状态口径（已确认）

- 这组状态描述的是“研发采购链路里的物料流转状态”，不等同于采购单、验收单、领料单自己的审核状态。
- `待采购`：小仓管理员已填写并保存采购表单，等待采购角色确认接单。
- `采购中`：采购角色已确认接单，采购执行中（或待供应商/到货）。
- `取消`：采购被驳回，或因业务原因终止，不再继续后续流转。
- `到货`：实物已到主仓，主仓可据此对照采购信息核货。
- `验收`：主仓已完成验收，并在合适时点将物料交接/转入小仓。
- `领取`：物料已在小仓侧被项目或项目式归集项领用。
- `报废`：物料在主仓或小仓链路中被判定报废，结束流转。
- `退回`：物料走公司级退回处理，仍由主仓统一承接，不要求小仓管理员直接操作。

## 方案候选

### 方案 A：受限子仓模型（推荐）

- 摘要：在同一平台内引入“主仓 + 研发小仓”仓别语义，但第一阶段只开放主仓到小仓的内部调拨，以及小仓自己的出库、盘点、报废、报表。
- 补充：同时引入“研发采购需求 -> 采购执行 -> 主仓按采购信息验收”的协同链路，采购信息服务于主仓验收自动带出。
- 补充：研发采购到货后的推荐库存路径为“主仓验收入主仓 -> 按实际交接再转小仓”，避免因时间差造成库存责任失真。
- 代价：中
- 风险：低
- 优点：
  - 最贴合当前已确认业务口径：小仓独立运转，但不承担公司级外部入出。
  - 小仓管理员工作台边界清晰，权限与报表容易按仓别裁剪。
  - 能复用 `inventory-core`、`project`、`rbac`、`reporting` 现有方向。
- 缺点：
  - 需要给库存、单据、报表增加仓别维度。
  - 需要引入主仓到小仓的专门协同单据或明确调拨语义。
- 复用点：`inventory-core`、`project`、`rbac`、`reporting`

### 方案 B：项目库存池增强模型

- 摘要：不真正引入小仓仓别，而是把研发小仓视为“项目归集库存池 + 权限视图”，主仓领料后直接转成项目库存，再由小仓页面操作。
- 代价：低
- 风险：高
- 优点：
  - 改动可能更小，更多复用现有 `project` 语义。
  - 第一版开发速度可能更快。
- 缺点：
  - 很难自然表达“小仓像独立小仓一样运转”的现实。
  - 盘点、报废、库存流水、报表容易和真实仓库语义拧巴。
  - 后续若继续扩展别的小仓，模型会越来越别扭。
- 复用点：`project`、`reporting`、`rbac`

### 方案 C：通用多仓框架

- 摘要：直接把系统升级成通用多仓架构，主仓、研发小仓以及未来其它仓位都放入统一仓库体系，配套更完整的仓间调拨、仓别报表和权限模型。
- 代价：高
- 风险：中
- 优点：
  - 长期最完整，未来扩展更多仓别最自然。
  - 模型统一，后续能力上限高。
- 缺点：
  - 超出当前“主仓 + 一个研发小仓”的最小闭环。
  - 会把第一阶段范围、数据迁移、前后端改动面都显著放大。
- 复用点：`inventory-core`、`rbac`，但需要更大范围扩展

## 最终确认方案

最终确认选择 **方案 A：受限子仓模型**。

原因：它既承认“研发小仓已经真实独立运转”，又守住“公司级入出统一经过主仓”的管理原则；相比直接上通用多仓，范围更稳；相比仅做项目库存池，又更符合你当前的真实业务。
