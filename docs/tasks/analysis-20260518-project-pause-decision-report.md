# 项目暂停 / 继续决策分析报告

- 日期：2026-05-18
- 项目：Saifute WMS / 旧系统迁移 / 库存重算 / 月度报表
- 目的：给管理层判断“是否暂停项目”提供事实依据，避免把历史数据问题、人工核验成本、报表性能问题，事后简单归因成“开发没有提醒”。

## 一句话结论

不建议把当前问题简单定性为“技术做不出来”。更准确的判断是：

> 系统功能主体已经做出不少，但历史数据质量、库存价格层、月报金额口径和报表性能进入了收口阶段。这个阶段不是靠继续写代码就能线性推进，必须有仓库 / 财务 / 管理层参与确认历史事实。

如果老板的目标是“先停掉投入”，可以暂停；但暂停前必须书面接受以下后果：

1. 当前历史数据问题不会因为暂停而消失。
2. 未来再启动时，旧库和目标库的数据边界可能继续漂移，恢复成本会更高。
3. 月报慢、金额不一致、历史价格层对不上等问题，仍需要逐条核验和处理。
4. 如果没有业务人员确认历史单据真假，技术团队不能替业务拍板“哪条历史数据是真实的”。

更稳妥的建议是：**暂停新增功能，但保留一个短周期的“数据核验 + 报表性能收口”决策窗口**。窗口结束后，再由老板决定完全暂停、继续上线，还是只保留核心业务不做历史报表承诺。

## 资料边界说明

本报告依据当前仓库内已经留下的任务文档、迁移报告、修复报告和代码路径整理。需要特别说明：

- 不同报告来自不同日期的执行截面，例如 2026-05-10 / 2026-05-11 的全量导入与 replay 记录、2026-05-17 的车间领料价格层漂移记录。
- 如果旧库或目标库在报告之后继续录单、修数据、重新 replay，行数和 warning 数可能变化。
- 因此，老板做正式暂停 / 继续决策前，必须先确认采用哪一个数据快照作为决策依据。

这也是本报告建议先做短周期“冻结数据边界”的原因。否则后续容易出现“当时说的数字”和“后来看到的数字”不一致，双方都难以解释。

### 2. 历史数据问题是真实存在的，而且不是纯技术判断题

最近的车间领料价格层漂移分析显示：

- 扫描到 `78` 行车间领料 / 报废单据存在“单据成本价”和“库存来源价格层”不一致。
- `34` 行已自动修复。
- `44` 行已通过单独核验 / 专项确认 / 说明字段拆分 / 物料定向重建修复。
- 当前剩余阻断为 `0` 行。
- 最后一次通用复扫结果为 `planned=0, blocked=0`。

这类问题不能直接批量强改。原因很简单：

- 如果单据价格错了，应修正单据成本口径。
- 如果单据价格对，但缺少同价入库来源，应先补齐或修复入库 / 期初来源。
- 如果直接改库存流水价格，会破坏库存来源链，后续报表可能更难解释。

### 3. 月报慢的问题有技术因素，但目前还缺少性能验收标准

月报物料分类视角目前不是只查一张表：

- 当月入库 / 销售 / 车间领退料行要参与统计。
- 月初、月末库存数量和金额要从 `inventory_log` 回算。
- 物料汇总前端有默认 `50` 条分页，但这是展示分页；后端仍要先算完整汇总结果。

也就是说，前端分页只能减少页面一次展示多少行，不能根治后端聚合慢。

从代码上看，月初 / 月末库存回算会对 `inventory_log.biz_date <= 月末日期` 的流水进行按物料分组汇总。随着历史流水增加，这类查询天然会变重。如果老板要求“报表必须秒开”，就需要明确性能目标，例如：

- 页面查询：3 秒内、5 秒内，还是 10 秒内可接受。
- Excel 导出：是否允许异步生成。
- 月初 / 月末金额：是否允许用月结快照，而不是每次从全量流水回算。

如果没有这个标准，团队只能反复“感觉慢、继续改”，容易变成无期限投入。

### 4. 月报功能已做，但长期月报闭环没有完全结束

月度报表需求里已经明确：第一阶段重点是“仓库侧本期发生金额”，不是先证明所有历史期初 / 期末余额完全可信。

当前已完成的能力包括：

- 按自然月、业务日期统计总入 / 总出 / 净发生。
- 领域视角：入库、车间、销售、研发项目、RD 小仓。
- 物料分类视角。
- 系统内查看和 Excel 导出。

但仍然不能混淆：

- “能生成报表”不等于“历史所有金额都已经财务级确认”。
- “replay validate 通过”不等于“历史所有负库存、错价、无来源单据都已经业务确认”。
- “前端分页了”不等于“报表查询性能已经达标”。

## 暂停项目的影响

### 暂停的好处

1. 可以立即停止继续投入，避免在历史烂账上无限追加成本。
2. 可以避免在数据未确认前承诺上线，降低后续扯皮风险。
3. 可以倒逼管理层明确：到底要“可用系统”，还是要“把旧账完全审清楚”。

### 暂停的风险

1. 历史问题不会自动关闭，未来重启仍要处理。
2. 旧库如果继续被使用，数据边界会继续变化，后续再迁移会重新对账。
3. 当前已经形成的迁移脚本、修复规则、人工确认上下文会随时间遗失。
4. 月报和库存价格层问题如果不形成冻结清单，后面很难判断是老问题还是新问题。
5. 业务人员可能以为“暂停 = 问题由技术消化”，但实际需要仓库 / 财务确认的事项仍未消失。

## 不暂停继续做的影响

### 继续的好处

1. 现有迁移、库存重算、月报链路可以继续收口，不浪费已投入成果。
2. 最近的问题已经从“大面积未知”收敛到“有报告、有清单、有修复路径”的状态。
3. 可以用短周期把性能、数据核验、上线边界定下来。

### 继续的风险

1. 如果没有仓库 / 财务配合，开发仍会卡在“这条历史单据到底对不对”。
2. 如果老板要求历史报表完全准确，但不给人工核验资源，周期会继续拉长。
3. 如果继续边开发边改历史数据，没有冻结边界，报表和库存结果会反复变化。

## 建议老板做的决策

建议不要只问“暂停还是继续”，而是明确选择下面三种模式之一。

| 选项 | 内容 | 适合情况 | 代价 |
| --- | --- | --- | --- |
| A. 完全暂停 | 停止开发、迁移、报表优化，只保留当前资料 | 老板决定短期不再投入 | 历史问题保留；未来重启成本增加 |
| B. 有限收口后再暂停 | 暂停新增功能，只做数据清单、性能基线、风险报告、冻结快照 | 老板想控制成本，但不想留下烂尾争议 | 需要 1 个短周期投入 |
| C. 继续推进上线 | 继续修复数据、优化报表、做验收 | 老板接受需要业务核验资源 | 必须指定仓库 / 财务责任人和验收标准 |

本人建议选择 **B. 有限收口后再暂停 / 决策**。

原因：当前直接完全暂停，会留下大量“以后解释不清”的问题；但继续无限开发，也不合理。最稳妥的是先做一个短周期收口，让老板拿到可决策的清单。

## 建议的短周期收口范围

建议周期：`5 ~ 10` 个工作日。

目标不是继续做新功能，而是形成管理层可签字的结论。

### 收口 1：冻结数据边界

必须明确：

- 旧库以哪一天、哪一次备份为准。
- 目标库以哪一天、哪一次 replay / 修复为准。
- 暂停期间旧系统如果继续录单，后续是否还要再迁。

交付物：

- 源库备份路径和 hash。
- 目标库备份路径。
- 当前目标库关键表行数。
- 未解决数据问题清单。

### 收口 2：报表性能基线

必须测出真实耗时，而不是只说“慢”。

至少记录：

- 月报领域视角接口耗时。
- 月报物料分类视角接口耗时。
- Excel 导出耗时。
- 慢 SQL 或主要查询路径。
- 当前前端分页是否只是展示分页。

交付物：

- 一份性能报告。
- 明确优化方案：索引、月结快照、异步导出、缓存或限制查询范围。
- 老板确认的性能目标。

### 收口 3：历史数据核验清单

必须把“技术能修”和“必须人工确认”分开。

当前至少包含：

- 初始 `44` 行车间领料价格层漂移已全部闭环。
- 最终负库存盘点 warning。
- 0 成本、无来源、乱序、后续来源冲抵等历史例外。
- 已处理 `033`、`yh1`、`jg6`、`yf12`、`lb004`、`lb005`、`lb007`、`yh10`、`zjq106`、`yf42`、`zjq042`、`cp032`、`zjq149`、`yf81`、`yf2`、`cp001`、`yf57`、`004`；后续重点转为最终负库存 warning、0 成本、无来源和乱序历史例外。

交付物：

- 每条问题的单据号、物料、数量、金额、阻断原因。
- 建议处理方式：修单据、补来源、盘点调整、接受历史 warning。
- 需要仓库 / 财务确认的责任人。

### 收口 4：上线边界声明

老板需要明确选择：

1. 是否允许“历史遗留问题以 warning + 盘点调整收口”。
2. 是否要求历史报表达到财务审计级准确。
3. 是否允许只保证上线后新增业务严格正确，历史数据按核验清单逐步处理。
4. 月报是否允许标注“历史数据待核验”。

如果老板选择“历史全部准确后再上线”，那必须同步接受更长周期和人工核验成本。

## 责任边界建议

为了避免后续互相怪责，建议把责任边界写清楚：

| 事项 | 技术团队责任 | 业务 / 仓库 / 财务责任 | 管理层责任 |
| --- | --- | --- | --- |
| 迁移脚本 | 提供可重复执行、可验证、可回滚的脚本和报告 | 提供旧库含义、确认异常单据 | 确认是否允许清库 / 重建 / 切换 |
| 历史数据错误 | 找出差异、给出证据、提供修复工具 | 判断历史单据真实与否 | 指派确认人和确认时限 |
| 库存价格层 | 按单据和来源链重算，避免伪造价格层 | 确认无来源、错价、负库存如何处理 | 确认是否接受盘点调整收口 |
| 月报性能 | 测量、定位、优化查询和导出 | 确认常用查询范围 | 给出可接受的耗时标准 |
| 上线决策 | 说明技术和数据风险 | 说明业务可接受程度 | 最终决定上线、延期或暂停 |

## 建议给老板确认的话

建议直接让老板确认以下内容：

> 当前项目不是单纯“开发慢”。主要阻力已经进入历史数据核验、库存来源链修复、报表性能收口阶段。技术可以继续提供脚本、报告和修复工具，但不能替仓库和财务确认历史单据真假。如果现在暂停，需要接受后续重启成本和未处理历史问题；如果继续，需要明确核验人员、性能目标和上线边界。

老板需要选择：

- 是否完全暂停。
- 是否允许先做 `5 ~ 10` 个工作日的收口报告再决定。
- 是否安排仓库 / 财务人员参与核验。
- 是否接受历史 warning 通过盘点调整逐步收口。
- 月报性能目标是多少。
