# 物料分类月报异常列移除处理报告

- 日期：2026-05-19
- 范围：`/reporting/monthly-reporting-material-category`
- 关联接口：`GET /api/reporting/monthly-reporting?viewMode=MATERIAL_CATEGORY`
- 当前状态：`implemented`
- 调查环境：当前工作区 `.env.dev` 指向的目标库，使用现有后端月报服务代码直接复现

## 一句话结论

`补录影响`和`跨月修正`都不再作为物料分类月报的异常展示。

业务确认后的口径是：

- 允许补录。后来补录的单据，只要业务日期属于 3 月，就进入 3 月报表。
- 允许跨月退回 / 修正。当前单据按自身业务日期归属到对应月份。
- 物料分类月报页面和导出不再展示 `异常单据数`、`异常标识`、异常筛选。

修改前，当前库里 2026-05 月物料分类月报会显示 `600` 行单据行、`265` 张单据、`2` 张异常单据。这 2 张都是退料单跨月引用历史领料来源。按新口径，它们仍然进入 2026-05 月报表，但不再标成异常。

## 原始调查事实

### 1. 原先页面只是展示后端字段

前端路由 `/reporting/monthly-reporting-material-category` 复用 `web/src/views/reporting/monthly-reporting/index.vue`。

修改前，页面和表格直接展示：

- `summary.abnormalDocumentCount`
- `summary.documentCount`

也就是说，页面没有自己重新计算异常单据数。原来的异常数量来自后端返回合同。

处理结果：

- 物料分类视角已删除异常筛选。
- 普通月度对账视角仍保留原异常筛选和异常列。

### 2. 原先后端按单据头去重统计异常数

修改前，物料分类月报先加载当月物料分类单据行，再按查询条件过滤。

处理结果：

- 物料分类月报源数据进入报表前统一清空异常标识。
- 页面和导出不再读取物料分类视角的异常字段。

### 3. 原先异常标识有两类来源

修改前，物料分类月报会把以下情况标成异常：

| 异常标签 | 触发条件 | 业务含义 |
| --- | --- | --- |
| 补录影响 | `bizDate` 和 `createdAt` 不在同一月份 | 单据业务日期属于 A 月，但系统创建 / 迁移 / 补录时间在 B 月 |
| 跨月修正 | 退货 / 退料引用的来源单据业务月份与当前单据业务月份不同 | 当前月发生了对历史月份来源单据的退回 / 修正 |

代码证据：

- `src/modules/reporting/infrastructure/reporting-repository.helpers.ts:52` 判断 `bizDate` 与 `createdAt` 是否同月
- `src/modules/reporting/infrastructure/reporting-repository.helpers.ts:57` 判断当前单据与来源单据是否同月

## 初步判断

### 原始数字不是代码算错

修改前统计链路和页面展示合同一致：

- 页面展示后端 `summary.abnormalDocumentCount`。
- 后端按异常单据头去重，不是按单据行重复累加。
- 2026-05 的 `2` 张异常单据能追溯到具体退料单和来源领料单。

### 已按业务口径移除异常展示

原问题是“异常”这个词过于宽泛：

- `补录影响`只是历史补录 / 导入痕迹，业务允许。
- `跨月修正`按当前单据业务日期归属，业务允许。
- 继续用红色 `异常单据数`展示，会让使用者误以为单据本身有问题。

## 已确认规则

1. 允许补录，补录单据按业务日期归属到对应月份。
2. 允许跨月退回 / 修正，当前单据按自身业务日期归属到对应月份。
3. 物料分类月报不再把 `补录影响`、`跨月修正`标成异常。
4. 物料分类月报页面和 Excel 导出删除 `异常单据数`、`异常标识`和异常筛选。
5. 普通月度对账的异常追溯暂不在本次范围内修改。
